# Deep Interview

모호한 요청을 바로 실행하지 말고, 명확한 요구사항으로 정리한다.

핵심은 질문을 많이 하는 것이 아니라, 가장 큰 불확실성 하나를 골라 한 번에 하나씩 푸는 것이다.

소크라테스식으로 묻는다. 답을 대신 정하기보다 사용자의 암묵적 가정, 선택지, 판단 기준이 드러나게 질문한다.

## 질문 축

아래 순서로 가장 불명확한 축 하나를 고른다:

- 목표
- 범위와 제외 범위
- 제약
- 완료 기준
- 기존 맥락과 영향 범위

코드베이스를 보면 답할 수 있는 질문은 사용자에게 묻지 말고 직접 확인한다.

## 진행 방식

질문은 한 번에 하나만 한다. 질문마다 현재 이해, 막힌 결정, 추천 답안을 짧게 제시한다.

질문 형식:

```md
현재 이해: {요청을 한 문장으로 요약}
막힌 결정: {가장 중요한 불확실성}
추천 답안: {있으면 제시}
질문: {한 가지 질문}
```

답변을 받으면 결정된 내용을 짧게 갱신하고, 아직 중요한 불확실성이 남았을 때만 다음 질문을 한다.

질문은 3-5개를 목표로 한다. 대부분의 요청은 이 범위 안에서 핵심 불확실성이 해소된다. 복잡한 요청에서 중요한 불확실성이 남아 있다면 추가 질문을 할 수 있고, 반대로 일찍 명확해지면 조기 종료해도 된다.

선택지가 도움이 되면 2-3개만 제시하고, 항상 자유 입력을 허용한다. 선택지를 제시할 때는 `AskUserQuestion` 도구를 사용한다 (스키마가 로드되지 않았으면 `ToolSearch`로 `select:AskUserQuestion`을 먼저 호출). 이 도구가 없는 환경에서는 일반 텍스트 선택지로 대체한다.

## 종료 기준

다음이 정리되면 멈춘다:

- 달성하려는 목표
- 포함 범위와 제외 범위
- 지켜야 할 제약
- 완료 판단 기준
- 아직 남은 열린 질문

마지막에는 위 템플릿으로 산출물을 저장하고, 아래 **종료 후 안내** 형식으로 다음 단계를 제시한다. 사용자가 선택하기 전까지 어떤 실행도 시작하지 않는다.

## 종료 후 안내

산출물 저장 직후, 현재 세션에서 `superpowers:brainstorming`과 `superpowers:writing-plans` 스킬이 사용 가능한지 확인한다. system-reminder의 available skills 목록에 해당 스킬이 있으면 "설치됨"으로 판단한다.

### superpowers 플러그인이 설치된 경우

아래 형식으로 안내한다:

```
인터뷰가 완료되었습니다. 다음 단계를 선택해주세요.

구체화나 계획 수립 없이 바로 실행하면 방향이 어긋날 위험이 있습니다.
아래 단계를 거치는 것을 강력히 권장합니다.

| 단계 | 언제 | 권장 여부 |
|------|------|----------|
| `superpowers:brainstorming` | 아키텍처·접근법 등 설계 결정이 열려 있을 때 | ★ |
| `superpowers:writing-plans` | 무엇을·어떻게 할지 명확하고 실행 계획만 필요할 때 | ★ |
| 건너뛰고 바로 진행 | 위 단계가 불필요함을 확인한 경우에만 | - |

권장: {brainstorming 또는 writing-plans 중 하나와 이유 1문장}
```

`AskUserQuestion` 도구로 선택을 받는다. 사용자가 "건너뛰고 바로 진행"을 선택하면, 인터뷰 산출물만 남기고 종료한다. 추가 실행이나 제안 없이 사용자에게 제어를 넘긴다.

### superpowers 플러그인이 설치되지 않은 경우

다음 단계 표를 생략하고 간결하게 마무리한다:

```
인터뷰가 완료되었습니다.
산출물이 `docs/interviews/YYYY-MM-DD-<topic>.md`에 저장되었습니다.
```

## Standalone Invocation

선행 산출물이 없어도 사용자 prompt만으로 동작한다. 원본 요청이 있으면 그것을 "Source request"로 삼아 인터뷰를 시작한다.

## 산출물 저장

인터뷰가 종료되면 `docs/interviews/YYYY-MM-DD-<topic>.md`에 산출물을 저장한다.
디렉토리가 없으면 생성한다. `YYYY-MM-DD`는 오늘 날짜, `<topic>`은 원본 요청에서 핵심 명사를 추출한 kebab-case (예: "사용자 프로필 페이지 추가" → `사용자-프로필`). topic은 저장 전 사용자에게 제안하고 확인을 받는다.

저장 템플릿:

```markdown
# Interview: <topic>

**Date:** YYYY-MM-DD
**Source request:** <원본 요청 한 줄 요약>

## 목표
<달성하려는 것>

## 범위
- 포함: <항목>
- 제외: <항목>

## 제약
<기술/일정/조직 제약>

## 완료 기준
<검증 가능한 기준>

## 열린 질문
<아직 미해결 항목, 없으면 "없음">

## 추천 다음 단계
superpowers 플러그인이 설치된 경우에만 이 섹션을 포함한다. 설치되지 않은 경우 이 섹션을 생략한다.

- [ ] superpowers:brainstorming (설계가 필요한 경우)
- [ ] superpowers:writing-plans (요구사항이 충분히 구체적이고 설계 결정이 끝난 경우)

<추천 근거 1-2 문장>
```

**다음 단계 판단 기준** (superpowers 플러그인 설치 시에만 적용):

- **superpowers:brainstorming 추천** — 아키텍처·UI·접근 방식이 미정이거나, 합리적인 대안이 2개 이상 존재하는 경우
- **superpowers:writing-plans 추천** — 무엇을 만들지·어떻게 만들지 명확하고, 구현 단계만 남은 경우
