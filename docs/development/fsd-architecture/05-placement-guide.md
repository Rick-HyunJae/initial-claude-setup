# 코드 배치 의사결정 가이드

## Pages First 원칙 (FSD v2.1)

**"코드는 실제로 재사용이 필요할 때 하위 레이어로 이동한다."**

처음부터 features로 추출하지 않는다.  
한 곳에서만 쓰인다면 page나 widget에 두는 것이 정상이다.

---

## 배치 의사결정 트리

```
새 코드를 어디에 놓을까?
│
├─ 전역 설정, Provider, 라우터인가?
│   └─ YES → app/
│
├─ 딱 한 페이지에서만 쓰이는가?
│   └─ YES → pages/{페이지명}/
│            (UI, 상태, API 모두 해당 page 안에서 관리)
│
├─ 여러 페이지에서 쓰이는 UI 블록인가? (헤더, 사이드바 등)
│   └─ YES → widgets/{블록명}/
│
└─ 여러 곳에서 재사용된다면 → 아래 기준으로 세분화
    │
    ├─ 비즈니스 로직 없는 인프라인가?
    │   (UI 컴포넌트, 유틸 함수, API 클라이언트, 라우트 상수)
    │   └─ YES → shared/
    │
    └─ 재사용되는 사용자 행동 단위인가?
        ("사용자가 ~할 수 있다"로 표현됨)
        (로그인, 좋아요, 장바구니 추가, 댓글 작성)
        └─ YES → features/{행동명}/
```

## 상황별 배치 예시

### 케이스 1: 프로필 페이지의 프로필 폼

| 상황                                | 배치                                   |
| ----------------------------------- | -------------------------------------- |
| 프로필 페이지에서만 쓰임            | `pages/profile/ui/ProfileForm`         |
| 프로필 + 설정 페이지 두 곳에서 쓰임 | `features/profile-edit/ui/ProfileForm` |

### 케이스 2: 상품 카드 컴포넌트

| 상황                                 | 배치                                      |
| ------------------------------------ | ----------------------------------------- |
| 상품 목록 페이지에서만 쓰임          | `pages/product-list/ui/ProductCard`       |
| 여러 페이지에서 재사용되는 도메인 UI | `features/product-display/ui/ProductCard` |
| 비즈니스 무관한 일반 카드 레이아웃   | `shared/ui/Card`                          |

### 케이스 3: 상품 데이터 조회

| 상황                        | 배치                                    |
| --------------------------- | --------------------------------------- |
| 상품 상세 페이지에서만 사용 | `pages/product-detail/api/fetchProduct` |
| 여러 페이지에서 상품 조회   | `features/product-list/api/productApi`  |

### 케이스 4: 인증 관련 코드

| 코드                             | 배치             |
| -------------------------------- | ---------------- |
| 로그인 폼 + 로그인 로직          | `features/auth/` |
| 현재 유저 상태 (세션)            | `shared/auth/`   |
| HTTP 클라이언트 (axios 인스턴스) | `shared/api/`    |

### 케이스 5: 모달 관리

| 코드                                | 배치                       |
| ----------------------------------- | -------------------------- |
| 모달을 띄우는 인프라 (ModalManager) | `shared/ui/modal-manager/` |
| 모달 안에 들어가는 특정 내용        | 해당 page나 feature 안에   |

## 추출 시점 판단 기준

재사용이 필요한 시점이 추출 시점이다.

| 시그널                                   | 행동                           |
| ---------------------------------------- | ------------------------------ |
| 같은 코드가 2개 이상의 page에서 사용됨   | `features/`로 이동             |
| 같은 UI 블록이 여러 page에 복사됨        | `widgets/`로 이동              |
| 같은 유틸 함수가 여러 feature에서 사용됨 | `shared/lib/`으로 이동         |
| 현재는 1곳에서만 사용됨                  | 그 자리에 둔다 (추출하지 않음) |

## 슬라이스 이름 짓기

- 기준: **비즈니스 행동 또는 도메인 개념**
- 형식: kebab-case
- 범위: 너무 넓지 않게 (하나의 책임)

```
# 좋은 슬라이스 이름
features/add-to-cart/
features/submit-review/
features/user-profile/

# 나쁜 슬라이스 이름 (너무 포괄적)
features/user-management/    → auth/, profile-edit/, password-reset/ 으로 분리
features/product-stuff/      → 의미 없는 이름
```

## Pages First가 맞는 이유

| 장점                 | 설명                                  |
| -------------------- | ------------------------------------- |
| 응집도 향상          | 관련 코드가 한 곳에 모여 파악이 쉬움  |
| 삭제 용이            | 기능 제거 시 한 폴더만 삭제           |
| 추상화 오버헤드 없음 | 불필요한 레이어 분리 없이 빠르게 개발 |
| 자연스러운 분해      | 실제 재사용 패턴이 보일 때 리팩토링   |
